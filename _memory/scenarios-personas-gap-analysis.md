## 1. 페르소나 5명

규모·숙련도·매입처·마켓·운영방식이 모두 달라야 누락 시나리오를 잡아낼 수 있어 의도적으로 분산.

### P1. 김초보 (Jin)
- **28세, 직장인 부업, 1년차**
- 쿠팡 위주, 월 20-30 주문
- 매입은 amazon US 만 (가전 액세서리 위주)
- IT 친숙도: 보통 / 영어 약함
- 동기: 부업으로 월 30만~50만 수익
- 페인포인트:
  - 매입 → 배송 흐름 헷갈림
  - 영어 form 입력 실수 잦음
  - 매번 같은 주소·통관코드 입력 귀찮음
  - 환불·취소 처리 어려움

### P2. 박풀타임 (Park)
- **42세, 전업 셀러, 5년차**
- 쿠팡 + 스마트스토어 + 옥션 + 11번가 다중
- 월 300-500 주문
- 매입: amazon US/JP + 라쿠텐 + 타오바오 4곳
- 다양한 카테고리 (가전·패션·생활용품·식품)
- 사용 배대지 5곳 이상 동시
- 동기: 본업, 가족 부양
- 페인포인트:
  - 매일 50-100건 일괄 처리 필요
  - 매칭 자동화 부족 시 시간 폭발
  - 배대지 정산·결제 사이클 관리
  - 부분 배송·합배송 처리
  - 환율 변동 → 마진 계산
  - 매입 결제 다양한 카드 (한도 분산)

### P3. 이브랜드 (Lee)
- **35세, 자사몰 + 인스타 + 스마트스토어, 3년차**
- 일본 제품 전문 (라쿠텐 90%, amazon JP 10%)
- 월 100-150 주문 (개당 단가 높음, 평균 30만원)
- 자체 브랜드 이미지 중요 — 깔끔한 포장·빠른 배송 강조
- IT 친숙도: 높음
- 동기: 자체 브랜드 확장, 한국 시장 자리잡기
- 페인포인트:
  - 일본어 → 한국어 + 영문 상품명 변환 매번
  - 라쿠텐 결제 자동입력 안 됨
  - 카테고리별 통관 (식품/화장품/전자제품 규제 다름)
  - 배대지 보관 기간 vs 회전율
  - 마켓 구매자 클레임 응대 채널

### P4. 최규모 (Choi)
- **38세, 법인 셀러, 7년차, 직원 5명**
- 월 1000+ 주문 (자사몰 50% + 쿠팡 30% + 기타 20%)
- 매입: amazon US/JP 위주 + 자체 매입 (직접 접촉)
- ERP 사용 중 (자체 개발 또는 Excel)
- IT 친숙도: 매우 높음, 직원이 운영 담당
- 동기: 운영 효율 / 데이터 기반 의사결정
- 페인포인트:
  - 짐스캐너 ↔ ERP 연동 안 됨 → 수동 데이터 동기화
  - 직원 권한 분리 X (다중 사용자 X)
  - API 미제공 → 자동화 불가
  - 부가세·세무 자료 (분기·연간)
  - 배대지 정산 차이 (실 청구 vs 예측)

### P5. 정노련 (Jung)
- **55세, 구매대행 → 셀러 전환 1세대, 8년차**
- 단순 일본 직구 (라쿠텐·아마존JP) + 카톡 주문 (개인 거래)
- 월 50-80 주문
- IT 친숙도: 낮음 / 한 화면에서 다 보고 싶음
- 동기: 안정적 본업, 기존 단골 유지
- 페인포인트:
  - 메뉴 많으면 헷갈림
  - 모바일에서 처리하고 싶음
  - 직접 종이 적어가며 처리 (전화 받으면서)
  - 자동화보단 신뢰 가능한 수동
  - 새로운 도구 학습 부담

## 2. 페르소나별 시나리오

각 페르소나의 **골든 패스 (이상 흐름)** + **분기 변형**.

### P1. 김초보 — 첫 사용 + 작은 워크플로우

#### 골든 패스 (G1)
1. signup 6 step 완료 (이메일·약관·사업자·연락처·인증)
2. 대시보드 진입 → 빈상태 가이드 봄 → "확장 설치" 안내
3. `/settings/extension` → 토큰 발급 → popup 에 저장
4. `/settings/forwarder-addresses` → 몰테일 NJ 공용 주소 클릭 → 본인 이름·회원번호 추가 → 기본 설정
5. 쿠팡에서 주문 1건 발생 (다이슨 청소기) → 알림 확인 (이메일/카톡)
6. `/orders/new` → 4 섹션 폼 작성 (마켓/구매자/매입/배대지)
7. amazon.com 에서 매입 시도 → checkout/.../address → 확장 [🏠] → 내 주소 클릭 → form 자동 채움 → 결제
8. amazon 영수증 발생 → 확장 자동 수집 (또는 your-orders 에서 [📦])
9. `/imports` 에서 영수증 봄 → 매칭 추천 클릭 → 주문에 연결
10. 며칠 후 몰테일에 도착 알림 → 몰테일 사이트 로그인
11. 배송신청서 페이지 → 확장 [🪄 자동 채우기] → 통합형 인식 → 주문 선택 → form 자동 채움
12. 배대지 배송비 결제 (몰테일 자체 결제)
13. 한국 도착 → 몰테일이 마켓 구매자 주소로 발송
14. `/orders` 상태 변경 → "구매자 수령" / "구매 확정"

#### 분기 (G1-V)
- **V1 (확장 토큰 발급 안 함)**: 영수증 자동 수집 X → /imports 비어있음 → 수동 영수증 등록 페이지 필요? (현재 없음)
- **V2 (본인 주소 등록 안 함)**: amazon checkout 에서 공용 주소만 보임 → 회원번호 없는 채로 결제 → 배대지에서 매입자 식별 실패 (큰 문제)
- **V3 (영수증만 등록, 주문 미등록)**: amazon 에서 호기심에 구매했는데 마켓 주문이 없음 → /imports 에 "주문 없음" 영수증 → 어떻게 처리?
- **V4 (한 주문에 매입 실패 + 재구매)**: 첫 매입 카드 거절 → 다른 카드로 재시도 → 영수증 2개 발생 → 어느 게 진짜인가?
- **V5 (잘못된 주문 등록 → 삭제)**: /orders/new 작성 후 실수 발견 → 주문 삭제 → b2b_orders.deleted_at 처리 → 영수증 매칭 자동 해제?

#### 발견 갭 (G1)
- ❌ 첫 onboarding tour 없음 (메뉴 어디 가야할지 불명확)
- ❌ 수동 영수증 등록 (확장 못 쓸 때 / 라쿠텐·타오바오) UI 없음
- ❌ "주문 없는 영수증" 처리 UX (지금은 미매칭 상태로 방치)
- ❌ 매입 실패 → 재시도 시 영수증 중복 처리
- ❌ 주문 삭제 시 매칭된 영수증 처리 정책 (자동 해제? cascade?)
- ❌ 확장 본인 phone fallback 시점 — 가입 직후엔 phone 없을 수도 (가입 step 에서 입력 강제?)

### P2. 박풀타임 — 일괄 처리·다중 매입처

#### 골든 패스 (G2)
1. 매일 아침 9시 — 마켓 주문 100건 발생
2. 각 마켓 어드민에서 주문 정보 Excel 다운로드 (4개 마켓)
3. `/orders/bulk` → 4번 paste → 100건 등록 (각 마켓별)
4. 점심 후 amazon US/JP + 라쿠텐 + 타오바오 매입 시작 (50건씩)
5. amazon → 확장 자동 영수증 수집 / 라쿠텐·타오바오 → **수동 등록 필요 (현재 미구현)**
6. /imports 100건 영수증 → /orders/matching 가서 한 번에 매칭 (탭으로 보고 추천 적용)
7. /products/matching 에서 N:N 매칭 (반복 매입 SKU)
8. 며칠 후 배대지 5곳에 분산 도착 → 각 배대지 사이트 들어가 신청서 작성 (확장 + 자동입력)
9. 배대지 결제 (한 주 1~2회 모아서)
10. 도착 → 마켓 구매자에게 배대지가 자동 발송 (몰테일 같은 큰 곳은 한국 택배 직접)
11. /orders 상태 변경 batch (delivered)

#### 분기 (G2-V)
- **V1 (라쿠텐 매입 수동 영수증)**: 라쿠텐 구매 후 영수증 받음 → /imports 어디서 수동 입력? (UI 없음)
- **V2 (일괄 매칭)**: 100건 영수증 × 100건 주문. 추천이 한 건당 1회 클릭이면 100번 클릭. 일괄 매칭 기능?
- **V3 (다중 매입처 한 주문)**: 마켓 1주문에 amazon US 1품 + 라쿠텐 1품 → 라인 2개. 각 라인이 다른 배대지로 가야 함. 현재 b2b_orders.forwarder_id 는 1개. → 라인별 forwarder 필요
- **V4 (다중 마켓 동일 SKU)**: 쿠팡 + 스마트스토어 둘 다에서 같은 SKU 주문 → 한 번에 amazon 에서 2개 매입 → 영수증 1개 × 주문 2개 매칭?
- **V5 (배대지 통합 후 한 박스)**: 같은 배대지에서 매주 한 박스로 합배송 → 정산은 박스 단위 → /orders 비용 계산?
- **V6 (반품)**: 마켓 구매자가 환불 신청 → 매입은 이미 진행됨 → 짐스캐너에서 환불 흐름 처리?
- **V7 (매입 가격 변경)**: 영수증 도착했는데 가격이 처음 등록한 unit_price_foreign 과 다름 → 자동 갱신? 수동 확인?

#### 발견 갭 (G2)
- ❌ 라쿠텐·야후·타오바오 확장 스크래퍼 미지원
- ❌ 수동 영수증 등록 UI (라쿠텐 등 영수증을 손으로 입력)
- ❌ 일괄 매칭 (100건 한 번에 추천 적용)
- ❌ 라인별 forwarder (현재 주문 1개에 forwarder 1개)
- ❌ 한 영수증 ↔ 여러 주문 매칭 (현재 1:N: 영수증 1개 - 주문 1개)
- ❌ 합배송 박스 단위 비용 정산
- ❌ 환불·취소 흐름 모델 (status enum 에 환불 단계 없음)
- ❌ 영수증 도착 후 unit_price diff alert
- ❌ 배대지별 정산·결제 사이클 추적 (수동)
- ❌ 운송장 자동 트래킹 (외부 API)

### P3. 이브랜드 — 일본 매입·통관·브랜드

#### 골든 패스 (G3)
1. 자사몰 주문 들어옴 → 수동으로 /orders/new 등록 (자사몰 API 미연동)
2. 라쿠텐에서 일본어 상품 검색 → 매입 → 라쿠텐 영수증 (수동 입력)
3. 일본어 상품명 → 한국어/영문 변환 (수동, 화장품은 영문 표기 필요)
4. /products 에 해외 상품 등록 (영문명 포함)
5. /products/domestic 에 자사몰용 한국 상품명 등록
6. /products/matching N:N
7. 일본 배대지 (오마이집 후쿠오카) → 본인 주소 등록 + 일본 phone
8. 라쿠텐 결제 시 자동입력 → **현재 라쿠텐 미지원**
9. 배대지 도착 → 배송신청서 (일본어 form) → 확장 자동입력
10. 통관 → 화장품/식품은 별도 신고 → 짐스캐너 가이드 없음
11. 한국 도착 → 자사몰 구매자에게 직접 발송 (셀러가 운영하는 택배)

#### 분기 (G3-V)
- **V1 (식품 통관 거부)**: 일본 식품 구매 → 한국 통관 거부 → 매입은 했는데 도착 안 함 → 짐스캐너 처리?
- **V2 (영문 상품명 매번 입력)**: 100개 라쿠텐 상품 → 100번 영문명 입력 → 번역 API?
- **V3 (라쿠텐 회원 등급)**: 라쿠텐 슈퍼 세일 / 포인트 → 영수증 금액 ≠ 표시 가격 → 매칭 점수 정확도 떨어짐
- **V4 (브랜드 정품 인증)**: 일본 정품 인증 라벨 → 한국 판매 시 첨부 → 상품에 메타데이터?
- **V5 (반품/AS)**: 화장품 알러지 환불 → 일본 매입처에 반품? 짐스캐너 흐름?

#### 발견 갭 (G3)
- ❌ 자사몰 API 연동
- ❌ 라쿠텐·야후 확장 스크래퍼 (영수증 + checkout 자동입력)
- ❌ 영문/한글 상품명 자동 번역
- ❌ 카테고리별 통관 안내 (식품·화장품·전자제품 등 KCS 가이드)
- ❌ 통관 거부 시 처리 흐름 (status 단계)
- ❌ 일본 amazon/라쿠텐 포인트·할인 영수증 처리
- ❌ 정품 인증·시리얼번호 메타 (b2b_products 확장)

### P4. 최규모 — 자동화·ERP 연동·다중 사용자

#### 골든 패스 (G4)
1. 자사몰 + 쿠팡 + 스마트스토어 주문 자동 수집 (현재 X — 수동 일괄)
2. 자체 ERP 와 짐스캐너 동기화 (현재 X)
3. 매입은 amazon 자동 수집 / 가격 비교 (현재 X)
4. 매칭 자동 (heuristic + 상품매칭 기반 SKU 매칭)
5. 직원 A: 주문 등록 / 직원 B: 매입 / 직원 C: 배대지 신청서 / 직원 D: 정산
6. 분기·연간 부가세 자료 자동 export
7. 매출·마진 대시보드 → 의사결정

#### 분기 (G4-V)
- **V1 (직원 권한)**: 직원마다 다른 권한 (열람만 vs 등록 vs 결제)
- **V2 (외부 API)**: 자체 ERP 가 짐스캐너 API 로 주문 PUSH / 영수증 GET
- **V3 (대량 환불)**: 단일 SKU 결함으로 50건 환불 → 일괄 처리
- **V4 (월말 정산)**: 5개 배대지 × 월간 정산 vs 청구서 검증
- **V5 (사업자 계정 추가)**: 법인 + 개인 사업자 2개 운영 → 한 계정에서 둘 다?

#### 발견 갭 (G4)
- ❌ 다중 사용자/팀원 권한 (현재 1 user = 1 account)
- ❌ 마켓 API 자동 import
- ❌ 외부 API (REST) 제공
- ❌ 매입처 가격 비교
- ❌ 배대지 정산 자동 검증 (예측 vs 실 청구)
- ❌ 분기/연간 부가세 자료 (현재 월 단위만)
- ❌ 다중 사업자 계정 전환
- ❌ 대량 환불 일괄 처리
- ❌ 매입 결제 자동화 (카드 등록 후 자동)

## 3. Cross-cutting 시나리오 (페르소나 무관 공통)

### C1. 환불·취소
- **C1.1** 마켓 구매자 환불 (매입 전): 주문 취소 → 매입 안 함 → status='cancelled' → 쿼터 차감?
- **C1.2** 마켓 구매자 환불 (매입 후·발송 전): 매입 도착 → 재고로 → 다른 구매자에게? 폐기?
- **C1.3** 마켓 구매자 환불 (발송 후): 반품 받음 → 환불 처리 → 회계 영향
- **C1.4** 매입처 환불 (amazon defect): amazon 환불 → 영수증 status 변경?
- **C1.5** 부분 환불: 5개 중 1개 환불 → 영수증 액수 일부 변경

### C2. 배송 이슈
- **C2.1** 매입 분실 (운송 중)
- **C2.2** 배대지에서 분실
- **C2.3** 한국 통관 거부 (식품·화장품)
- **C2.4** 부분 도착 (5개 시켰는데 3개만 옴)
- **C2.5** 합배송 박스 분할 (배대지가 무게로 2박스 나눔)
- **C2.6** 운송장 미발급 / 추적 불가

### C3. 시간 흐름
- **C3.1** 주문 등록 후 며칠 뒤 매입 (이슈: 환율 변동)
- **C3.2** 매입 후 한 달 뒤 도착 (해외 직배 지연)
- **C3.3** 도착 후 보관 (배대지 보관일 한도 초과 → 폐기 또는 추가 비용)
- **C3.4** 시즌 (블랙프라이데이·연말 폭주 → 시스템 부하·배대지 지연)

### C4. 결제·정산
- **C4.1** 매입 결제 카드 변경 (한도 분산)
- **C4.2** 배대지 정산 청구 vs 예측 차이
- **C4.3** 환율 적용 시점 (매입일 vs 정산일 vs 부가세 신고일)
- **C4.4** 부분 결제 / 분할 결제 (amazon 의 split shipment)

### C5. 데이터 정합·이력
- **C5.1** 주문 수정 → 변경 이력 (audit log)
- **C5.2** 매입 영수증 재수집 (이미 있는 거 덮어쓰기 vs 신규)
- **C5.3** 배대지 양식 변경 (배대지가 form 개편) — 기존 매핑 무효화
- **C5.4** 회사 폐업·계정 폐쇄 → 데이터 보관 vs 삭제

### C6. 법규·세금
- **C6.1** 부가세 신고 (분기·연간)
- **C6.2** 통관코드 (개인통관부호) 미입력·잘못 입력
- **C6.3** 직구 vs 사입 (간이과세 vs 일반과세 셀러)
- **C6.4** 명의 도용 / 통관코드 도용 (마켓 구매자가 다른 사람 번호 도용)

### C7. 고객 응대 (마켓 구매자 입장)
- **C7.1** 배송 조회 — 마켓 구매자가 짐스캐너 추적 페이지 보고 싶음 (셀러 노출 없이)
- **C7.2** 클레임 — 마켓 구매자 → 셀러 → 짐스캐너 ↔ 배대지 ↔ 매입처
- **C7.3** Q&A — "왜 이렇게 오래 걸리나" 질문 → 셀러가 트래킹 보여줘야 함

## 5. 2차 검토 — 제3자/리뷰어 관점

다른 관점 (코드 모르는 외부 컨설턴트가 본다는 가정):

### 🔍 흐름 일관성
- "주문관리 > 해외 주문 목록" 과 "매입관리 > 해외주문관리" 두 메뉴가 같은 페이지 (/imports). 셀러 혼란.
- /orders/matching 과 /imports 내 매칭 UI 가 같은 작업을 두 군데서 함. 어느 게 권장 워크플로우?
- "해외 주문" 이라는 용어가 영수증 (amazon 결제 후) 인지 매입 요청 (결제 전) 인지 모호.

### 🔍 정보 위계
- 대시보드 통계가 어떤 의사결정에 쓰이는가? (현재: 주문/매출/SKU 카운트만)
- /orders 목록에서 한눈에 봐야 할 정보 (영수증 매칭 상태? 배대지 도착 여부? 마켓 구매자 응대 필요?)
- 우선처리 큐 (예: "지금 행동 필요한 주문 12건") 없음

### 🔍 데이터 모델
- 한 주문에 forwarder 1개 (한 라인은 한 배대지). 분기 G2-V3 (다중 매입처 한 주문) 처리 어려움
- b2b_order_items 의 supplier_order_number 가 영수증과 중복 (영수증의 supplier_order_number 와 같은 의미). 정규화 필요?
- 환불·취소 status 단계 enum 부재. 현재 enum: pending/confirmed/paid/forwarder_submitted/in_transit/arrived_korea/delivered/completed. cancelled/refunded/disputed 등 부재.

### 🔍 신뢰성·복구
- 영수증 자동 수집 실패 시 retry / 재시도 메커니즘 없음
- 확장이 폼 자동 채울 때 일부 selector 실패 → 사용자가 모름 (현재: 못 채운 필드 표시는 하지만 visibility 부족)
- 매칭 잘못된 경우 다시 풀기 가능하나, "왜 그 매칭이었는지" 이력 없음

### 🔍 비즈니스
- 짐스캐너의 수익 모델은? (구독·트랜잭션·affiliate?)
- 무료 vs 유료 플랜 기능 분기는?
- 셀러가 짐스캐너를 떠난다면 (churn) — 데이터 export?
- 매출 0건 셀러 (가입만 하고 안 씀) 활성화 흐름?

### 🔍 제3자가 봤을 때 즉시 의문
- "왜 /orders 와 /imports 가 별도 메뉴인가? 둘 다 주문이라며"
- "확장 없으면 짐스캐너 가치가 절반인가?"
- "한 주문에 매입 안 되면 어떻게 표시되는가? 그냥 pending 으로 영원히?"
- "배대지 신청서 자동입력이 안 되는 배대지 (selector 미수집) 면 셀러는 어떻게?"
