## 1. 페르소나 5명

규모·숙련도·매입처·마켓·운영방식이 모두 달라야 누락 시나리오를 잡아낼 수 있어 의도적으로 분산.

### P1. 김초보 (Jin)
- **28세, 직장인 부업, 1년차**
- 쿠팡 위주, 월 20-30 주문
- 매입은 amazon US 만 (가전 액세서리 위주)
- IT 친숙도: 보통 / 영어 약함
- 동기: 부업으로 월 30만~50만 수익
- 페인포인트:
  - 매입 → 배송 흐름 헷갈림
  - 영어 form 입력 실수 잦음
  - 매번 같은 주소·통관코드 입력 귀찮음
  - 환불·취소 처리 어려움

### P2. 박풀타임 (Park)
- **42세, 전업 셀러, 5년차**
- 쿠팡 + 스마트스토어 + 옥션 + 11번가 다중
- 월 300-500 주문
- 매입: amazon US/JP + 라쿠텐 + 타오바오 4곳
- 다양한 카테고리 (가전·패션·생활용품·식품)
- 사용 배대지 5곳 이상 동시
- 동기: 본업, 가족 부양
- 페인포인트:
  - 매일 50-100건 일괄 처리 필요
  - 매칭 자동화 부족 시 시간 폭발
  - 배대지 정산·결제 사이클 관리
  - 부분 배송·합배송 처리
  - 환율 변동 → 마진 계산
  - 매입 결제 다양한 카드 (한도 분산)

### P3. 이브랜드 (Lee)
- **35세, 자사몰 + 인스타 + 스마트스토어, 3년차**
- 일본 제품 전문 (라쿠텐 90%, amazon JP 10%)
- 월 100-150 주문 (개당 단가 높음, 평균 30만원)
- 자체 브랜드 이미지 중요 — 깔끔한 포장·빠른 배송 강조
- IT 친숙도: 높음
- 동기: 자체 브랜드 확장, 한국 시장 자리잡기
- 페인포인트:
  - 일본어 → 한국어 + 영문 상품명 변환 매번
  - 라쿠텐 결제 자동입력 안 됨
  - 카테고리별 통관 (식품/화장품/전자제품 규제 다름)
  - 배대지 보관 기간 vs 회전율
  - 마켓 구매자 클레임 응대 채널

### P4. 최규모 (Choi)
- **38세, 법인 셀러, 7년차, 직원 5명**
- 월 1000+ 주문 (자사몰 50% + 쿠팡 30% + 기타 20%)
- 매입: amazon US/JP 위주 + 자체 매입 (직접 접촉)
- ERP 사용 중 (자체 개발 또는 Excel)
- IT 친숙도: 매우 높음, 직원이 운영 담당
- 동기: 운영 효율 / 데이터 기반 의사결정
- 페인포인트:
  - 짐스캐너 ↔ ERP 연동 안 됨 → 수동 데이터 동기화
  - 직원 권한 분리 X (다중 사용자 X)
  - API 미제공 → 자동화 불가
  - 부가세·세무 자료 (분기·연간)
  - 배대지 정산 차이 (실 청구 vs 예측)

### P5. 정노련 (Jung)
- **55세, 구매대행 → 셀러 전환 1세대, 8년차**
- 단순 일본 직구 (라쿠텐·아마존JP) + 카톡 주문 (개인 거래)
- 월 50-80 주문
- IT 친숙도: 낮음 / 한 화면에서 다 보고 싶음
- 동기: 안정적 본업, 기존 단골 유지
- 페인포인트:
  - 메뉴 많으면 헷갈림
  - 모바일에서 처리하고 싶음
  - 직접 종이 적어가며 처리 (전화 받으면서)
  - 자동화보단 신뢰 가능한 수동
  - 새로운 도구 학습 부담

## 2. 페르소나별 시나리오

각 페르소나의 **골든 패스 (이상 흐름)** + **분기 변형**.

### P1. 김초보 — 첫 사용 + 작은 워크플로우

#### 골든 패스 (G1)
1. signup 6 step 완료 (이메일·약관·사업자·연락처·인증)
2. 대시보드 진입 → 빈상태 가이드 봄 → "확장 설치" 안내
3. `/settings/extension` → 토큰 발급 → popup 에 저장
4. `/settings/forwarder-addresses` → 몰테일 NJ 공용 주소 클릭 → 본인 이름·회원번호 추가 → 기본 설정
5. 쿠팡에서 주문 1건 발생 (다이슨 청소기) → 알림 확인 (이메일/카톡)
6. `/orders/new` → 4 섹션 폼 작성 (마켓/구매자/매입/배대지)
7. amazon.com 에서 매입 시도 → checkout/.../address → 확장 [🏠] → 내 주소 클릭 → form 자동 채움 → 결제
8. amazon 영수증 발생 → 확장 자동 수집 (또는 your-orders 에서 [📦])
9. `/imports` 에서 영수증 봄 → 매칭 추천 클릭 → 주문에 연결
10. 며칠 후 몰테일에 도착 알림 → 몰테일 사이트 로그인
11. 배송신청서 페이지 → 확장 [🪄 자동 채우기] → 통합형 인식 → 주문 선택 → form 자동 채움
12. 배대지 배송비 결제 (몰테일 자체 결제)
13. 한국 도착 → 몰테일이 마켓 구매자 주소로 발송
14. `/orders` 상태 변경 → "구매자 수령" / "구매 확정"

#### 분기 (G1-V)
- **V1 (확장 토큰 발급 안 함)**: 영수증 자동 수집 X → /imports 비어있음 → 수동 영수증 등록 페이지 필요? (현재 없음)
- **V2 (본인 주소 등록 안 함)**: amazon checkout 에서 공용 주소만 보임 → 회원번호 없는 채로 결제 → 배대지에서 매입자 식별 실패 (큰 문제)
- **V3 (영수증만 등록, 주문 미등록)**: amazon 에서 호기심에 구매했는데 마켓 주문이 없음 → /imports 에 "주문 없음" 영수증 → 어떻게 처리?
- **V4 (한 주문에 매입 실패 + 재구매)**: 첫 매입 카드 거절 → 다른 카드로 재시도 → 영수증 2개 발생 → 어느 게 진짜인가?
- **V5 (잘못된 주문 등록 → 삭제)**: /orders/new 작성 후 실수 발견 → 주문 삭제 → b2b_orders.deleted_at 처리 → 영수증 매칭 자동 해제?

#### 발견 갭 (G1)
- ❌ 첫 onboarding tour 없음 (메뉴 어디 가야할지 불명확)
- ❌ 수동 영수증 등록 (확장 못 쓸 때 / 라쿠텐·타오바오) UI 없음
- ❌ "주문 없는 영수증" 처리 UX (지금은 미매칭 상태로 방치)
- ❌ 매입 실패 → 재시도 시 영수증 중복 처리
- ❌ 주문 삭제 시 매칭된 영수증 처리 정책 (자동 해제? cascade?)
- ❌ 확장 본인 phone fallback 시점 — 가입 직후엔 phone 없을 수도 (가입 step 에서 입력 강제?)

### P2. 박풀타임 — 일괄 처리·다중 매입처

#### 골든 패스 (G2)
1. 매일 아침 9시 — 마켓 주문 100건 발생
2. 각 마켓 어드민에서 주문 정보 Excel 다운로드 (4개 마켓)
3. `/orders/bulk` → 4번 paste → 100건 등록 (각 마켓별)
4. 점심 후 amazon US/JP + 라쿠텐 + 타오바오 매입 시작 (50건씩)
5. amazon → 확장 자동 영수증 수집 / 라쿠텐·타오바오 → **수동 등록 필요 (현재 미구현)**
6. /imports 100건 영수증 → /orders/matching 가서 한 번에 매칭 (탭으로 보고 추천 적용)
7. /products/matching 에서 N:N 매칭 (반복 매입 SKU)
8. 며칠 후 배대지 5곳에 분산 도착 → 각 배대지 사이트 들어가 신청서 작성 (확장 + 자동입력)
9. 배대지 결제 (한 주 1~2회 모아서)
10. 도착 → 마켓 구매자에게 배대지가 자동 발송 (몰테일 같은 큰 곳은 한국 택배 직접)
11. /orders 상태 변경 batch (delivered)

#### 분기 (G2-V)
- **V1 (라쿠텐 매입 수동 영수증)**: 라쿠텐 구매 후 영수증 받음 → /imports 어디서 수동 입력? (UI 없음)
- **V2 (일괄 매칭)**: 100건 영수증 × 100건 주문. 추천이 한 건당 1회 클릭이면 100번 클릭. 일괄 매칭 기능?
- **V3 (다중 매입처 한 주문)**: 마켓 1주문에 amazon US 1품 + 라쿠텐 1품 → 라인 2개. 각 라인이 다른 배대지로 가야 함. 현재 b2b_orders.forwarder_id 는 1개. → 라인별 forwarder 필요
- **V4 (다중 마켓 동일 SKU)**: 쿠팡 + 스마트스토어 둘 다에서 같은 SKU 주문 → 한 번에 amazon 에서 2개 매입 → 영수증 1개 × 주문 2개 매칭?
- **V5 (배대지 통합 후 한 박스)**: 같은 배대지에서 매주 한 박스로 합배송 → 정산은 박스 단위 → /orders 비용 계산?
- **V6 (반품)**: 마켓 구매자가 환불 신청 → 매입은 이미 진행됨 → 짐스캐너에서 환불 흐름 처리?
- **V7 (매입 가격 변경)**: 영수증 도착했는데 가격이 처음 등록한 unit_price_foreign 과 다름 → 자동 갱신? 수동 확인?

#### 발견 갭 (G2)
- ❌ 라쿠텐·야후·타오바오 확장 스크래퍼 미지원
- ❌ 수동 영수증 등록 UI (라쿠텐 등 영수증을 손으로 입력)
- ❌ 일괄 매칭 (100건 한 번에 추천 적용)
- ❌ 라인별 forwarder (현재 주문 1개에 forwarder 1개)
- ❌ 한 영수증 ↔ 여러 주문 매칭 (현재 1:N: 영수증 1개 - 주문 1개)
- ❌ 합배송 박스 단위 비용 정산
- ❌ 환불·취소 흐름 모델 (status enum 에 환불 단계 없음)
- ❌ 영수증 도착 후 unit_price diff alert
- ❌ 배대지별 정산·결제 사이클 추적 (수동)
- ❌ 운송장 자동 트래킹 (외부 API)

### P3. 이브랜드 — 일본 매입·통관·브랜드

#### 골든 패스 (G3)
1. 자사몰 주문 들어옴 → 수동으로 /orders/new 등록 (자사몰 API 미연동)
2. 라쿠텐에서 일본어 상품 검색 → 매입 → 라쿠텐 영수증 (수동 입력)
3. 일본어 상품명 → 한국어/영문 변환 (수동, 화장품은 영문 표기 필요)
4. /products 에 해외 상품 등록 (영문명 포함)
5. /products/domestic 에 자사몰용 한국 상품명 등록
6. /products/matching N:N
7. 일본 배대지 (오마이집 후쿠오카) → 본인 주소 등록 + 일본 phone
8. 라쿠텐 결제 시 자동입력 → **현재 라쿠텐 미지원**
9. 배대지 도착 → 배송신청서 (일본어 form) → 확장 자동입력
10. 통관 → 화장품/식품은 별도 신고 → 짐스캐너 가이드 없음
11. 한국 도착 → 자사몰 구매자에게 직접 발송 (셀러가 운영하는 택배)

#### 분기 (G3-V)
- **V1 (식품 통관 거부)**: 일본 식품 구매 → 한국 통관 거부 → 매입은 했는데 도착 안 함 → 짐스캐너 처리?
- **V2 (영문 상품명 매번 입력)**: 100개 라쿠텐 상품 → 100번 영문명 입력 → 번역 API?
- **V3 (라쿠텐 회원 등급)**: 라쿠텐 슈퍼 세일 / 포인트 → 영수증 금액 ≠ 표시 가격 → 매칭 점수 정확도 떨어짐
- **V4 (브랜드 정품 인증)**: 일본 정품 인증 라벨 → 한국 판매 시 첨부 → 상품에 메타데이터?
- **V5 (반품/AS)**: 화장품 알러지 환불 → 일본 매입처에 반품? 짐스캐너 흐름?

#### 발견 갭 (G3)
- ❌ 자사몰 API 연동
- ❌ 라쿠텐·야후 확장 스크래퍼 (영수증 + checkout 자동입력)
- ❌ 영문/한글 상품명 자동 번역
- ❌ 카테고리별 통관 안내 (식품·화장품·전자제품 등 KCS 가이드)
- ❌ 통관 거부 시 처리 흐름 (status 단계)
- ❌ 일본 amazon/라쿠텐 포인트·할인 영수증 처리
- ❌ 정품 인증·시리얼번호 메타 (b2b_products 확장)

### P4. 최규모 — 자동화·ERP 연동·다중 사용자

#### 골든 패스 (G4)
1. 자사몰 + 쿠팡 + 스마트스토어 주문 자동 수집 (현재 X — 수동 일괄)
2. 자체 ERP 와 짐스캐너 동기화 (현재 X)
3. 매입은 amazon 자동 수집 / 가격 비교 (현재 X)
4. 매칭 자동 (heuristic + 상품매칭 기반 SKU 매칭)
5. 직원 A: 주문 등록 / 직원 B: 매입 / 직원 C: 배대지 신청서 / 직원 D: 정산
6. 분기·연간 부가세 자료 자동 export
7. 매출·마진 대시보드 → 의사결정

#### 분기 (G4-V)
- **V1 (직원 권한)**: 직원마다 다른 권한 (열람만 vs 등록 vs 결제)
- **V2 (외부 API)**: 자체 ERP 가 짐스캐너 API 로 주문 PUSH / 영수증 GET
- **V3 (대량 환불)**: 단일 SKU 결함으로 50건 환불 → 일괄 처리
- **V4 (월말 정산)**: 5개 배대지 × 월간 정산 vs 청구서 검증
- **V5 (사업자 계정 추가)**: 법인 + 개인 사업자 2개 운영 → 한 계정에서 둘 다?

#### 발견 갭 (G4)
- ❌ 다중 사용자/팀원 권한 (현재 1 user = 1 account)
- ❌ 마켓 API 자동 import
- ❌ 외부 API (REST) 제공
- ❌ 매입처 가격 비교
- ❌ 배대지 정산 자동 검증 (예측 vs 실 청구)
- ❌ 분기/연간 부가세 자료 (현재 월 단위만)
- ❌ 다중 사업자 계정 전환
- ❌ 대량 환불 일괄 처리
- ❌ 매입 결제 자동화 (카드 등록 후 자동)

## 3. Cross-cutting 시나리오 (페르소나 무관 공통)

### C1. 환불·취소
- **C1.1** 마켓 구매자 환불 (매입 전): 주문 취소 → 매입 안 함 → status='cancelled' → 쿼터 차감?
- **C1.2** 마켓 구매자 환불 (매입 후·발송 전): 매입 도착 → 재고로 → 다른 구매자에게? 폐기?
- **C1.3** 마켓 구매자 환불 (발송 후): 반품 받음 → 환불 처리 → 회계 영향
- **C1.4** 매입처 환불 (amazon defect): amazon 환불 → 영수증 status 변경?
- **C1.5** 부분 환불: 5개 중 1개 환불 → 영수증 액수 일부 변경

### C2. 배송 이슈
- **C2.1** 매입 분실 (운송 중)
- **C2.2** 배대지에서 분실
- **C2.3** 한국 통관 거부 (식품·화장품)
- **C2.4** 부분 도착 (5개 시켰는데 3개만 옴)
- **C2.5** 합배송 박스 분할 (배대지가 무게로 2박스 나눔)
- **C2.6** 운송장 미발급 / 추적 불가

### C3. 시간 흐름
- **C3.1** 주문 등록 후 며칠 뒤 매입 (이슈: 환율 변동)
- **C3.2** 매입 후 한 달 뒤 도착 (해외 직배 지연)
- **C3.3** 도착 후 보관 (배대지 보관일 한도 초과 → 폐기 또는 추가 비용)
- **C3.4** 시즌 (블랙프라이데이·연말 폭주 → 시스템 부하·배대지 지연)

### C4. 결제·정산
- **C4.1** 매입 결제 카드 변경 (한도 분산)
- **C4.2** 배대지 정산 청구 vs 예측 차이
- **C4.3** 환율 적용 시점 (매입일 vs 정산일 vs 부가세 신고일)
- **C4.4** 부분 결제 / 분할 결제 (amazon 의 split shipment)

### C5. 데이터 정합·이력
- **C5.1** 주문 수정 → 변경 이력 (audit log)
- **C5.2** 매입 영수증 재수집 (이미 있는 거 덮어쓰기 vs 신규)
- **C5.3** 배대지 양식 변경 (배대지가 form 개편) — 기존 매핑 무효화
- **C5.4** 회사 폐업·계정 폐쇄 → 데이터 보관 vs 삭제

### C6. 법규·세금
- **C6.1** 부가세 신고 (분기·연간)
- **C6.2** 통관코드 (개인통관부호) 미입력·잘못 입력
- **C6.3** 직구 vs 사입 (간이과세 vs 일반과세 셀러)
- **C6.4** 명의 도용 / 통관코드 도용 (마켓 구매자가 다른 사람 번호 도용)

### C7. 고객 응대 (마켓 구매자 입장)
- **C7.1** 배송 조회 — 마켓 구매자가 짐스캐너 추적 페이지 보고 싶음 (셀러 노출 없이)
- **C7.2** 클레임 — 마켓 구매자 → 셀러 → 짐스캐너 ↔ 배대지 ↔ 매입처
- **C7.3** Q&A — "왜 이렇게 오래 걸리나" 질문 → 셀러가 트래킹 보여줘야 함

## 5. 2차 검토 — 제3자/리뷰어 관점

다른 관점 (코드 모르는 외부 컨설턴트가 본다는 가정):

### 🔍 흐름 일관성
- "주문관리 > 해외 주문 목록" 과 "매입관리 > 해외주문관리" 두 메뉴가 같은 페이지 (/imports). 셀러 혼란.
- /orders/matching 과 /imports 내 매칭 UI 가 같은 작업을 두 군데서 함. 어느 게 권장 워크플로우?
- "해외 주문" 이라는 용어가 영수증 (amazon 결제 후) 인지 매입 요청 (결제 전) 인지 모호.

### 🔍 정보 위계
- 대시보드 통계가 어떤 의사결정에 쓰이는가? (현재: 주문/매출/SKU 카운트만)
- /orders 목록에서 한눈에 봐야 할 정보 (영수증 매칭 상태? 배대지 도착 여부? 마켓 구매자 응대 필요?)
- 우선처리 큐 (예: "지금 행동 필요한 주문 12건") 없음

### 🔍 데이터 모델
- 한 주문에 forwarder 1개 (한 라인은 한 배대지). 분기 G2-V3 (다중 매입처 한 주문) 처리 어려움
- b2b_order_items 의 supplier_order_number 가 영수증과 중복 (영수증의 supplier_order_number 와 같은 의미). 정규화 필요?
- 환불·취소 status 단계 enum 부재. 현재 enum: pending/confirmed/paid/forwarder_submitted/in_transit/arrived_korea/delivered/completed. cancelled/refunded/disputed 등 부재.

### 🔍 신뢰성·복구
- 영수증 자동 수집 실패 시 retry / 재시도 메커니즘 없음
- 확장이 폼 자동 채울 때 일부 selector 실패 → 사용자가 모름 (현재: 못 채운 필드 표시는 하지만 visibility 부족)
- 매칭 잘못된 경우 다시 풀기 가능하나, "왜 그 매칭이었는지" 이력 없음

### 🔍 비즈니스
- 짐스캐너의 수익 모델은? (구독·트랜잭션·affiliate?)
- 무료 vs 유료 플랜 기능 분기는?
- 셀러가 짐스캐너를 떠난다면 (churn) — 데이터 export?
- 매출 0건 셀러 (가입만 하고 안 씀) 활성화 흐름?

### 🔍 제3자가 봤을 때 즉시 의문
- "왜 /orders 와 /imports 가 별도 메뉴인가? 둘 다 주문이라며"
- "확장 없으면 짐스캐너 가치가 절반인가?"
- "한 주문에 매입 안 되면 어떻게 표시되는가? 그냥 pending 으로 영원히?"
- "배대지 신청서 자동입력이 안 되는 배대지 (selector 미수집) 면 셀러는 어떻게?"

## 12. Adoption Funnel · TTV (Time to Value)

가입 ~ 첫 자동화 효과 보기까지 시간. 어디서 이탈하는가.

### 단계별 funnel (P1 김초보 기준)

```
[가입 시작] (100%)
  ↓ signup step 1: 이메일·약관 (성공률 95%)
[step 1 완료] (95%)
  ↓ step 2-4: 사업자 인증·연락처 (75% — 사업자등록 없는 셀러 25% 이탈)
[step 4 완료] (71%)
  ↓ step 5: 사업자 진위 확인 (NTS 미연동 — 셀러는 그냥 통과) (95%)
[가입 완료] (67%)
  ↓ 첫 로그인 → 대시보드 (90%)
[첫 로그인] (60%)
  ↓ "어디서 시작" 모름 → 1주일 안 들어옴 (40% 이탈)
[1주일 후 재방문] (36%)
  ↓ /orders/new 첫 주문 등록 (성공률 80% — 폼 마찰)
[첫 주문 등록] (29%)
  ↓ 확장 설치 / 토큰 (50% — 모르거나 안 함)
[확장 활성] (14%)
  ↓ 첫 매입 + 영수증 자동 수집 (90%)
[첫 자동화 가치 실현] (13%)
```

**TTV 추정**:
- 가입 ~ 첫 주문 등록: **평균 1주일** (등록만 한 채로 며칠 방치)
- 첫 주문 ~ 확장 자동화 가치 실현: **추가 1-2주일** (확장 설치·토큰·매입 흐름 거쳐야 함)
- 최종 TTV: **2-3주일** ← 너무 김

**Drop-off 지점 top 3**:
1. signup step 2 사업자 인증 (개인 셀러 25% 이탈)
2. 가입 후 1주일 idle (40% 이탈) — onboarding 부재
3. 확장 설치 안내 부재 (50% 이탈)

**개선 시 잠재 효과**: TTV 를 1주일 → 1-2일로 단축하면 전환율 13% → 30-40% 가능

## 13. 신뢰성·정확도 위협 시나리오

자동화가 잘못됐을 때 셀러의 반응 + 복구 흐름. 신뢰는 한 번 깨지면 회복 어렵다.

### 🚨 위협 T1: 영수증 매칭 잘못 (다른 주문에 연결)

**시나리오**: 추천 점수가 92점이라 클릭. 사실은 다른 마켓 구매자의 주문. 배송신청서 자동 채울 때 잘못된 buyer 정보가 amazon 영수증과 매칭돼 다른 구매자 주소로 발송됨.

**결과**: 구매자 A 가 받아야 할 게 구매자 B 에게 감. B 는 "이게 뭐냐" → 셀러가 회수·재발송 비용. 마켓 클레임. 평점 하락.

**짐스캐너 책임**: 추천이 잘못된 게 짐스캐너 잘못? 셀러가 확인 안 한 게 잘못? 모호함.

**현재 대응**:
- 매칭 시 confirmation 없음 — 1클릭 즉시 매칭
- 잘못된 매칭 후 [해제] 가능하지만 이미 신청서 작성 되었으면 늦음

**필요 보강**:
- 매칭 시 "확인 모달" (구매자 + 상품 + 금액 비교 1초)
- 매칭 이력·롤백 (audit log)
- 매칭 점수 < 90 면 "검증 권장" 경고

### 🚨 위협 T2: 확장 자동 채움 셀렉터 변경 → 잘못된 필드에 채움

**시나리오**: 짐패스가 form 개편. 기존 `input[name="recipientName"]` 가 `input[name="receiverName"]` 으로 변경. 확장의 휴리스틱이 다른 input (예: 송장 메모) 에 buyer_name 채움.

**결과**: 잘못된 필드에 한국 이름 들어감. 배대지에서 작업자가 보고 의아해함. 배송 지연.

**현재 대응**:
- 셀렉터 변경 자동 감지 X
- "못 채운 필드" 표시는 있지만, "잘못된 필드 채움" 표시는 없음

**필요 보강**:
- 폼 캡쳐 ([📋]) 누적해서 변경 감지
- 자동 채움 후 셀러가 시각적 확인할 수 있게 강조 (채워진 필드 노란 highlight)
- 셀렉터 fallback 매트릭스 (1순위 안 되면 2순위)

### 🚨 위협 T3: 영수증 자동 수집이 split shipment 누락

**시나리오**: amazon 한 주문이 2박스로 나눠 도착. 영수증 페이지에 박스1 만 표시. 확장이 박스1 영수증만 수집. 박스2 는 짐스캐너에 없음 → 마켓 주문 매칭 시 합계 안 맞음.

**결과**: 매칭 점수가 낮음 → 셀러가 "왜 매칭이 안 되지?" 의심. 또는 잘못된 다른 주문에 매칭.

**필요 보강**:
- amazon orders 페이지에서 "split shipment" 감지 + 모두 수집
- 한 supplier_order_number 에 여러 영수증 row 허용

### 🚨 위협 T4: 환율 변동 후 마진 계산 잘못

**시나리오**: 영수증 currency=USD, total_foreign=$200. 매입 시 환율 1380원. 짐스캐너에 한 달 후 부가세 신고할 때 1320원으로 변동. 셀러는 잘못된 환율로 계산.

**현재**: 환율 변동 알림 있지만 "어느 시점 환율을 써야 하는가" 명확하지 않음.

**필요 보강**:
- 영수증에 매입 시점 환율 저장 (snapshot)
- 부가세 신고 시 신고 시점 환율 별도 표시

### 🚨 위협 T5: 마켓 주문 등록 후 마켓에서 환불 → 짐스캐너는 모름

**시나리오**: 쿠팡 구매자가 환불 신청 (배송 전). 마켓 어드민에는 cancelled 상태. 짐스캐너에는 pending. 셀러가 환불 모르고 매입 진행.

**필요 보강**:
- 마켓 API 자동 sync (지금은 수동 입력)
- 또는 셀러가 일괄 status 변경 batch UI

### 🚨 위협 T6: 데이터 손실 (실수로 주문 삭제·매칭 해제)

**시나리오**: 셀러가 잘못된 주문이라 생각해서 삭제. 사실은 유효. 매칭된 영수증도 자동 해제됨. 영수증은 살아있지만 미매칭 상태.

**필요 보강**:
- soft delete (이미 deleted_at 있음. 활용 잘 되는지 확인)
- 휴지통 / 30일 보관 / 복구 UI

### 신뢰 회복 비용

신뢰 한 번 깨지면 셀러는 짐스캐너 일부 자동화 거부하고 수동 회귀.
- 매칭 자동 추천 신뢰 깨지면 → 매번 수동 매칭 (월 22시간 손실)
- 폼 자동 채움 신뢰 깨지면 → 매번 수동 (월 33-55시간 손실)
- 영수증 자동 수집 신뢰 깨지면 → 매번 수동 입력

**=> 신뢰 보강이 신규 기능보다 ROI 높을 수 있음.**

## 15. Edge Case 빈도·심각도 매트릭스

희귀하지만 심각한 케이스 / 자주 발생하지만 사소한 케이스 분류. 매트릭스 = (빈도 × 심각도).

### 매트릭스

| Edge Case | 빈도 (월/페르소나) | 심각도 (1-5) | 빈도×심각도 | 현재 대응 |
|---|---|---|---|---|
| amazon split shipment | 5-10건/월 | 3 (매칭 오류) | 30 | ❌ 미감지 |
| amazon 일부 취소 | 2-3건/월 | 3 | 9 | ❌ |
| 매입 카드 거절 → 재시도 | 3-5건/월 | 2 | 10 | ❌ 중복 영수증 가능 |
| 마켓 환불 (배송 전) | 5-10건/월 | 4 | 40 | ❌ 매뉴얼 status |
| 마켓 환불 (배송 후) | 1-2건/월 | 5 | 10 | ❌ |
| 배대지 통관 거부 | 1건/분기 | 5 | 5 | ❌ |
| 배대지에서 분실 | 1건/년 | 5 | 5 | ❌ 매뉴얼 |
| 배대지 부분 도착 | 2-3건/월 | 3 | 9 | ❌ |
| 배대지 form 개편 (selector 변경) | 1-2건/분기 | 4 | 8 | ⚠️ [📋] 캡쳐로 보강 가능 |

### 매트릭스 분석

**빈도×심각도 ≥ 30 인 case (즉시 대응 필요)**:
1. **마켓 환불 (배송 전)** — 40점. 발생률 높음 + 회계 영향
2. **통관코드 오류** — 40점. 구매자가 잘못 입력 → 통관 실패 → 배송 지연
3. **한 주문 다중 매입처** — 40점. 데이터 모델 자체 한계
4. **amazon split shipment** — 30점. 매칭 오류 유발
5. **모바일 처리** — 60점. 매일 발생하는 마찰

빈도×심각도 매트릭스로 보면 Part 1 의 우선순위와 좀 다름:
- **마켓 환불 처리**가 1순위로 올라옴 (Part 1 에서는 status enum 확장 정도)
- **통관코드 검증** 이 새로 등장 (Part 1 에선 안 다룸)
- **모바일 처리** 가 빈도×심각도 1위 — UX 자체 개선 큼

## 35. 셀러 감정·인지부하 (Emotional Journey)

각 단계에서 셀러가 느끼는 마음 상태. UX 디자인의 진짜 핵심.

### 감정 차원 정의

| 감정 | 의미 | 짐스캐너 영향 |
|---|---|---|
| 😀 만족 | "잘 됐다" | 재사용·추천 |
| 😐 무덤덤 | 기능 작동 (당연) | 평균 |
| 😟 불안 | "이게 맞나" | 검증·재확인 |
| 😤 좌절 | "왜 안 되지" | 수동 회귀·이탈 |
| 😱 충격 | 데이터 손실·잘못된 자동화 | churn 트리거 |
| 🤔 호기심 | "이 기능 뭐지" | 학습 기회 |
| 😎 자신감 | 익숙해진 자신 | sticky |

### P1 김초보 — 첫 매입 감정 곡선 (N1 기반)

```
19:30 카톡 알림 봄         → 🤔 호기심 + 😀 (첫 매출 흥분)
19:35 쿠팡 어드민 확인      → 😐 무덤덤 (익숙해진 작업)
19:38 짐스캐너 대시보드     → 🤔 "오늘 1건 카드 클릭 안 되네" 의문
19:40 /orders/new 폼       → 😟 불안 (필드 12개·통관코드 외움)
19:43 통관코드 빨강 검증    → 😱 "어, 틀렸나" → 😀 "다행히 즉시 알려주네"
19:48 등록 완료 토스트     → 😀 만족
19:52 amazon checkout      → 😟 "영문 form 어떻게 채우지"
19:53 확장 [🏠] 버튼 발견   → 😱 "어 이런 게 있었네" → 🤔 호기심
19:53 form 자동 채움       → 😀😀 큰 만족 (시간 절약 경험)

```

**감정 곡선 결론**: 처음 마찰 (불안·의문) → 확장 발견 → 큰 만족 → 자신감. 한 번 성공하면 sticky. **첫 사용 성공률이 critical**.

### P1 김초보 — 본인 주소 미등록 감정 (N2 기반)

```

몰테일 사이트 봤더니 없음   → 😟 "어디 갔지"
상담원: "회원번호 X"        → 😱 충격 (매물 분실 위기)
"본인 회원번호 다시"        → 😤 좌절 + 자책
1시간 상담 끝에 매물 회수   → 😀 안도 + 😤 (시간 낭비)
짐스캐너 본인 주소 등록     → 😐 (학습된 무덤덤)
다음 매입 — 본인 주소 사용 → 😎 자신감
```

**감정 곡선 결론**: 충격 → 좌절 → 학습. **첫 실수 후 학습 비용이 큼**. 사전 경고 가 있었으면 충격 회피.

**감정 곡선 결론**: 매번 같은 좌절 → 자동화 못 받으면 셀러 본업 지속 어려움. **번역 자동화가 ROI 가장 큼**.

### P5 정노련 — 새 기능 학습 거부 (N9 기반)


**감정 곡선 결론**: 안정성·예측성 > 새 기능. **변경 시 명확한 안내·교육 필수**. 새 기능을 강요하지 말고 옵션으로.

### P6 강신생 — 가입 후 좌절·이탈 감정 (N10 기반)

```
가입 1일차 대시보드 빈 화면    → 😟 "뭐부터 하지"
메뉴 둘러봄                    → 🤔 "배대지가 뭐지" (모르는 단어)
무엇부터 시작할지 모름           → 😤 좌절
2-5일차 idle                   → 😐 (잊어버림)
6일차 다시 들어옴               → 😟 (여전히 모름)
7일차 1:1 문의                 → 😐 답변 대기
답변 받음 "주문 들어오면"        → 😱 "그럼 매출 0인 나는 못 쓰네" → churn
```

**감정 곡선 결론**: **빈 대시보드 = 첫 좌절 트리거**. TTV 가 너무 길어 churn. **첫 30분 안에 가치 체험** 필수.

```
│   │   ├── 검색해서 정확한 주문 발견 → ✓
│   │   └── 검색해도 없음
│   │       ├── 주문 아직 등록 안 함 → /orders/new 등록 → 매칭
│   │       └── 영수증이 다른 셀러 거 (잘못 도착) → ⚠️
├── 추천 점수 < 50
│   ├── 셀러가 [🔍 검색]
│   │   ├── 발견 → ✓
│   │   ├── 미발견 → 주문 등록 또는 보류
│   │   └── 영수증이 "구매자 없는" 매입 (셀러 본인용) → 별도 처리?
│   └── 추천 자체 없음
│       └── 마찬가지로 [🔍 검색] 또는 보류
└── 영수증이 이미 매칭됨
    ├── [변경] → 다른 주문 검색
    ├── [해제] → 다시 미매칭 상태로
    └── 다른 영수증이 같은 주문에 매칭 시도 (1:N) → 현재 N:1 만 가능 ⚠️
```

**갭 발견**:
- 추천 점수 50-89 구간에서 셀러의 의심·검증 흐름
- 점수 < 50 또는 추천 없음 시 수동 검색
- "구매자 없는" 영수증 (셀러 본인 매입) 처리
- 1:N 매칭 (한 주문에 여러 영수증)

### Tree-3: 마켓 환불 신청 (P1, C2)

```
START [마켓 어드민 환불 신청]
├── 배송 전
│   ├── 매입 안 함 → /orders 삭제 → ✓
│   ├── 매입 진행 중 → amazon 매입 취소 시도
│   │   ├── amazon 취소 가능 (24h 이내) → 취소 → ✓
│   │   └── 취소 불가 → 도착 후 처리
│   └── 매입 완료, 발송 전 → amazon 반품?
│       ├── 반품 가능 → 반품 요청 → 시간·비용
│       └── 반품 불가 → 손실 떠안기
├── 배송 중
│   └── 배송 도중 환불 → 도착 후 셀러가 반품 받기
├── 배송 후 (구매자 수령)
│   ├── 14일 이내 (마켓 정책)
│   │   ├── 구매자 반품 → 셀러가 매물 살리기 → 재판매?
│   │   └── 환불 후 처리
│   └── 14일 초과 → 협의 (마켓 정책)
└── 부분 환불
    ├── 다라인 중 1라인 환불 → 라인 단위 status
    └── 수량 부분 환불 (5개 중 2개 결함) → 수량 차감
```

**갭 발견**:
- 환불 단계 status 13가지 모델
- 라인 단위·수량 단위 부분 환불
- 매물 살리기 → 재판매 흐름 (재고 자동 등록)
- 반품 비용 vs 손실 비교 도구

### Tree-4: 배대지 신청서 자동 채움 ([🪄])

```
START [배대지 신청서 페이지]
├── 페이지 인식
│   ├── combined (배송+매입) → 둘 다 채움

│   ├── purchase_only → 매입만
│   └── none → 버튼 안 뜸 (잘못 인식)
├── 주문 선택

│   │   ├── 클릭 → fill 진행
│   │   └── 여러 매칭 → 어느 거 선택? UX 모호
│   └── 매칭된 주문 0건 → "매칭 먼저" 안내 → /orders/matching
├── Fill 실행
│   ├── 모든 필드 채움 → ✓
│   ├── 일부 필드 missing (selector 미지원)

│   │   └── 셀러가 모르고 그대로 제출 → 배대지 거부

│       ├── 셀러가 발견 → 수정
│       └── 셀러가 모르고 제출 → ⚠️
```

- 결제 거부 → 신청서 만료 후 재진행

## 37. 시나리오 체인 (인과관계)

한 사건이 다음 사건을 트리거. 시간순 체인.

### Chain-1: P1 김초보의 "한 실수의 연쇄"
```
N1 (첫 매입 성공)
  ↓ 자신감 ↑
[2주 후] amazon 매입 5건 → 영수증 자동 수집
  ↓
N2 트리거 — 5건 모두 본인 주소 미등록 채로 결제
  ↓
배대지 5건 매물 식별 실패 → 매번 1시간 상담
  ↓ 좌절 누적
N3 트리거 — 다이슨 5건 중 1건 마켓 환불 신청
  ↓
짐스캐너 status 없어 internal_notes 메모
  ↓
[1달 후] 부가세 신고 — 환불 매출·매입 정합 모호
  ↓
짐스캐너 신뢰 ↓
  ↓
이탈 검토 또는 plan 다운그레이드
```

**핵심 인사이트**: **N2 (본인 주소 미등록)이 root cause** — 첫 매입 직후 안내 부재. 이후 모든 좌절 연쇄.

### Chain-2: P2 박풀타임의 "시즌·환불·번아웃"
```
[11월 1주차] 블프 준비 — 평소처럼
  ↓
[블프 당일] 주문 250건 폭주
  ↓
야근 + 배대지 결제 지연
  ↓
[블프 1주 후] 한국 도착 지연
  ↓
마켓 구매자 클레임 50건 → 환불 신청 30건
  ↓
짐스캐너 status 환불 단계 없음 → internal_notes 30건
  ↓
amazon 반품 시도 → 일부 30일 한도 초과
  ↓
손실 ~ 500만원
  ↓
[블프 2주 후] 정산 — 매출 vs 실수익 정합 어려움
  ↓
[블프 1달 후] 부가세 신고 — 환불 매출 회계 더 복잡
  ↓
번아웃 + plan 가치 의문
```

**핵심 인사이트**: 시즌 폭주는 **신뢰성 위협의 다발**. 시즌 알림·환불 status·자동 응대 함께 보강 필요.

### Chain-3: P3 이브랜드의 "통관 거부 → 브랜드 신뢰 ↓"
```
일본 식품 매입 100건
  ↓
[2주 후] 일부 통관 거부 (식품 한도)
  ↓
구매자 환불 안내 (카톡)
  ↓
구매자 인스타 후기 "신뢰 X — 환불 받음"
  ↓
브랜드 노출 감소
  ↓
다음 달 매출 30% 감소
  ↓
[2달 후] 식품 매입 자체 포기 → 라쿠텐 의존도 ↑ → 자동화 미지원 → 노동 ↑
```

**핵심 인사이트**: 통관 거부 한 번이 **브랜드 신뢰 → 매출 → 워크플로우** 까지 카스케이드. 카테고리별 통관 가이드 prevention 으로 큰 가치.

### Chain-4: P6 강신생 "가입 → idle → 이탈"
```
짐스캐너 가입 (1일차)
  ↓
대시보드 빈 화면 좌절
  ↓
1주 idle
  ↓
1:1 문의 → 답변 "매출 들어오면 사용" → 좌절
  ↓
이탈
  ↓
[1달 후] 진짜 첫 매출 발생 — 짐스캐너 기억 X
  ↓
경쟁사 도구 검색·사용
  ↓
짐스캐너 churn 확정
```

**핵심 인사이트**: 짐스캐너 가입 → 가치 실현 사이 **1달 공백**이 catastrophic. **데모 + 카운트다운·재방문 트리거** 필요.

### Chain-5: P4 최규모 "직원 매칭 오류 → 마켓 패널티"
```
직원 C 가 영수증 매칭 잘못 클릭 (10:00)
  ↓
배대지 신청서 [🪄] 다른 구매자 정보로 채움 (15:30)
  ↓
배대지 발송 → 다른 구매자 주소
  ↓
구매자 A "내 거 못 받음" 마켓 클레임 (3일 후)
  ↓
구매자 B "왜 다른 사람 거 받았지" 클레임
  ↓
마켓 두 건 패널티 (셀러 평점 하락)
  ↓
법인 매출 감소 (마켓 노출 순위 ↓)
  ↓
직원 C 책임 추궁 — 하지만 audit log 없음 → 본인 시인 X
  ↓
다음 달 다른 직원도 같은 실수 가능
```

**핵심 인사이트**: 다중 사용자 audit + confirm modal 부재가 **법인 차원 손실**. 직원 매칭 오류는 일반 셀러보다 조직 신뢰까지 영향.

## 40. 추가 narrative (Part 4 보완)

### N13. P2 박풀타임 — amazon 카드 거절 5단계 분기

**배경**: 박풀타임. 평일 10:30. amazon US 50건 매입 중 30번째 시도. **카드 거절**.

**흐름**:
1. amazon checkout [Place Order] → "Your payment was declined". 이유 안 알려줌.
2. 셀러: 신용카드 회사 (신한) 앱 들어가 확인 — "한도 초과 — 이번 달 사용량 1500만원 / 한도 1500만원".
3. 다른 카드로 변경 — 삼성카드. amazon 결제 → "처리 중" → "Order Placed". 영수증 발급. 카드 회사 OTP.
4. 확장 영수증 자동 수집 → /imports.
5. 매칭 추천 점수 91점. 클릭 매칭.
6. 31번째 시도 — 같은 다이슨 V15. **이미 카드 한도 다 썼는데** → 또 다른 카드 (KB).
7. KB 카드 OTP 폰에 안 옴. 5분 대기. 결제 만료. 다시 처음부터.
8. 결국 KB 카드 OTP 받음. 결제 완료.
9. 35번째 — 모든 카드 한도. amazon gift card 구매? amazon balance? 결정.
10. **amazon gift card $1000 구매** → balance → 35-40번째 결제. 영수증 금액 = gift card 사용분 = 다름 ($749 → $0 (gift card) 또는 $749 → $749 with note).
11. 41번째 — gift card 잔액 부족. 미루기.

**결과**: 50건 매입 → 41건 완료, 9건 다음 날로. 1시간 추가 손해.

**발견된 갭**:
- 매입 카드 한도 사전 알림 ("이번 달 amazon 한도 80% 사용 — 다음 매입 카드 부족 예상")
- 카드별 한도 추적 (셀러 본인 입력 — 짐스캐너에 카드 메타)
- amazon gift card 사용 시 영수증 정합 (gift card 결제 ≠ 카드 결제)
- 매입 진행 중 자동 카드 변경 (한 카드 거절 시 다른 카드 자동)

### N14. P3 이브랜드 — 라쿠텐 슈퍼세일 포인트 영수증

**배경**: 이브랜드. 12월 첫째 주 라쿠텐 슈퍼세일. 평소 ¥15,000 SK-II 가 ¥10,000 + 포인트 30% (¥3,000).

**흐름**:
1. 라쿠텐 슈퍼세일 SK-II 5개 매입. 결제 ¥50,000 → 카드 결제 + 포인트 ¥15,000 환급 예정.
2. 영수증 발급 — "결제액 ¥50,000 / 포인트 ¥15,000 / 실 비용 ¥35,000".
3. 짐스캐너에 라쿠텐 수동 영수증 등록 (UI 없음 → /orders/new 매입 정보에 ¥50,000 입력).
4. **포인트 환급은 1주 후 라쿠텐 계정에 잔액**. 다음 매입에 활용 가능.
5. 부가세 신고 — 매입 ¥50,000? ¥35,000? 회계사 문의.
6. 회계사: "실제 비용은 ¥35,000 — 포인트 환급 매출 차감"
7. 짐스캐너 데이터 ¥50,000 → 실제 ¥35,000 → **매번 수동 보정**.
8. 라쿠텐 다음 매입 — 포인트 사용. 결제 ¥10,000 (포인트 ¥10,000 + 카드 ¥0). 영수증 0 — 카드 결제 X.
9. 짐스캐너에 어떻게 등록? "0원 매입" — 인식 어려움.

**결과**: 회계 정합 매번 수동. 포인트 추적 어려움.

**발견된 갭**:
- 라쿠텐 포인트 처리 (할인·환급·사용 분리)
- 영수증 결제액 ≠ 실 비용 (포인트·할인) 모델
- 매입처 잔액·포인트 메타 (`b2b_supplier_balances` 테이블?)
- 환급액 매출 (수익) 표현
- 부가세 신고 시 실 비용 자동 계산

### N15. P7 한고가 — 가품 의심·분쟁 (6개월 후)

**배경**: 한고가. N11 의 6개월 후. 자사몰 구매자 (이XX)가 "감정 받으니 가품" 메일.

### N16. P8 도매준 — 도매 고객사 부도

**배경**: 도매준. 도매 고객사 C 가 8월에 향수 200개 발주. C 가 9월 부도. 미수금 ¥1,200,000.
