한 줄 평결

!
"엔진은 돌지만, 연료가 비어 있다"
변환 엔진·주문 파이프라인·확장 인프라는 실제로 동작한다. 하지만 팔리는 약속(33개 양식)의 콘텐츠가 1개뿐이고, 가입·이메일·결제가 외부 키 미비로 막혀 있어 현 상태로는 출시 불가.
1
MVP의 심장이 비어 있음 — '33 배대지 양식 자동 변환'인데 실제 공유 양식은 DB에 1개뿐 (≈3%)
변환 엔진(fillTemplate)·export API·Storage 원본은 실동작하지만, 라이브 DB의 공유 템플릿은 '짐패스 v1' 단 1행. 나머지 35개 배대지는 셀러가 직접 양식을 업로드하고 26여 개 컬럼을 손으로 매핑해야 함. 그런데 가입 랜딩·온보딩·요금제 등 5곳 이상이 "33/30+ 배대지 자동 변환"을 단정형으로 광고.
2
프로덕션 신규 가입 불가 — 사업자 진위확인(step-5)이 NTS 키 미등록으로 502 하드 게이트
NTS_BUSINESS_API_KEY 미설정 시 프로덕션에서 /api/verify-business가 502를 던지고, step-5는 건너뛰기 없는 필수 단계라 모든 신규 셀러가 가입을 완료할 수 없음. (개발 환경 mock은 아무 번호나 통과시켜 '잘 된다'고 착각하게 만듦.)
3
'수집 → 변환' 다리가 끊김 — 매칭된 영수증 데이터가 주문 라인으로 흘러들지 않음
브라우저 확장이 모은 영수증(현지주문번호·매입가)이 주문에 매칭돼도 b2b_order_items로 write-back 되지 않음. 매칭은 '연결·검증' 레이어일 뿐이라, 셀러는 양식에 들어갈 현지주문번호를 주문 등록 시 손으로 다시 입력(이중입력)해야 함.
엔드투엔드 워크플로우

셀러의 하루: 주문접수 → 해외매입 → 매칭 → 양식변환 → 배송·사후

사이트의 모든 메뉴는 이 5단계 가치사슬 위에 놓입니다. 변환 단계(④)가 핵심 가치이며, 그 앞뒤 메뉴들이 데이터를 채우고 사후를 관리합니다.

가치사슬 단절 지점
  • STEP 2 → 3 → 4: 수집한 영수증의 현지주문번호·매입가가 주문 라인으로 자동 반영되지 않아, 변환 양식 채움을 위해 같은 값을 두 번 입력해야 함 (이중입력).
  • STEP 4: 변환 엔진은 완성됐지만 공식 양식이 1개뿐 — 나머지 배대지는 셀러가 업로드 후 수작업 매핑(자동채움 0).
  • 가입 입구: NTS 키 미비로 step-5에서 가입이 막혀, 애초에 가치사슬에 진입하지 못할 수 있음.

최종 평결

출시까지 무엇을 먼저 채워야 하는가

고칠 곳은 코드가 아니라 대부분 콘텐츠·외부 키·결제입니다. 우선순위는 명확합니다.

목표 적합성: 핵심 가치는 옳고, 그것을 실현할 골격도 갖췄다.
"마켓 주문 1건 → 배대지 양식 30초 자동 변환"이라는 v0 MVP의 목표는 명확하고, 그 목표를 떠받치는 변환 엔진·주문 파이프라인·매칭 휴리스틱·확장 인프라·분석·보안까지 모두 실제로 구현되어 있습니다. 남은 것은 기능 추가가 아니라 "약속을 데이터로 채우는 일" — 양식 콘텐츠, 가입을 여는 NTS 키, 알림/고지를 보내는 RESEND 키, 결제를 받는 PG, 그리고 영문 번역과 수집→변환 데이터 다리입니다.
1
출시 차단 해제 (P0 — 이게 없으면 장사 자체가 안 됨)
① 수요 상위 5~10개 배대지의 공식 양식을 공유 템플릿으로 시드(또는 카피를 '현재 N개 지원'으로 정직하게 조정) · ② NTS_BUSINESS_API_KEY 등록 또는 step-5 임시 우회/수동심사 경로 추가 · ③ 매칭 시 영수증 값을 주문 라인으로 write-back(이중입력 제거).
BLOCKER × 3 — 가치사슬·가입·핵심양식
2
약속·법규 정합성 (P1 — 켜져 있다고 표기되는데 실제로 안 되는 것)
④ RESEND_API_KEY 등록 — 가입/정지/취소 안내 + 청약철회 자동고지(전자상거래법 제17조) 실발송(현재 통계 영구 0, 준법 리스크) · ⑤ 요금제의 결제 버튼·17% 할인 카피를 PG 연동 전까지 'Coming soon'으로 일치 · ⑥ 영문 상품명 번역(Papago) 연동 — 현재 alnum_only는 번역이 아니라 비ASCII 제거라 일·중 품명이 깨짐 · ⑦ 알림 producer 추가(현재 알림센터 항상 빈 상태).
HIGH — 표기·실제 불일치 / 외부 키
3
운영 견고성 & 사용성 (P2 — 둘째 달에 셀러를 잃지 않기)
⑧ 월 쿼터 리셋 cron 등록/검증(미가동 시 멀쩡한 셀러가 누적 사용량으로 차단) · ⑨ 운송장 입력이 광고하는 '한국행 자동 상태 전이' 실제 구현(현재 미동작) · ⑩ 주문 본문/라인 편집 API·UI 추가 · ⑪ 13개 상위 메뉴를 6~7개 그룹으로 정리 + 주문 상태 한글 라벨을 단일 소스로 통일(현재 6개 파일 중복·불일치) · ⑫ 추천 상품 mock → 실데이터, 매칭/행동큐 링크에 주문 컨텍스트 prefill.
MEDIUM — 운영 리스크 / 정보구조
